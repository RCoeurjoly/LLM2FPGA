* Task 3 RfP contract (2026-03-27)

** Task 3 engineering intent
- Task 3 must produce a fully transparent hardware flow. Temporary stubs or
  placeholder modules are acceptable only as explicit intermediate scaffolding
  with documented scope and a defined path to removal.
- Every operation that reaches the FPGA flow must have an explicit
  implementation by the time of the final Task 3 claim. Unsupported
  operations must be lowered, decomposed, rewritten, carried by an explicitly
  documented temporary stub, or rejected with a precise error. They must not
  be hidden behind opaque placeholders.
- Task 3 must not assume the full neural network fits on the FPGA at once.
  Model size may exceed FPGA capacity by a large factor, including 100x or
  more.
- When the model is larger than available fabric or tool capacity, Task 3 must
  still produce an explicit resource or bottleneck result rather than silently
  failing.
- For Task 3, an oversized-model/tool-capacity finding is acceptable if it is
  reproducible and tied to a clear artifact chain and command path.
- Partitioned execution strategy, temporal decomposition, and resource
  minimization belong to follow-on work unless Task 3 is explicitly redefined.
- Floating-point support must be fixed. Floating-point operations must either
  lower to a fully explicit floating-point implementation or lower to a fully
  explicit alternative numeric form with defined semantics. In both cases there
  must be no opaque fallback.
- Numeric transformations, including float-to-fixed or any other numeric
  conversion, must be deliberate, visible, and semantically documented. They
  cannot be hidden inside black-box components.
- Task 3 is successful only if the toolchain carries the model through to a
  reviewer-visible emitted-design result without unresolved black boxes or
  unresolved stubs, while preserving a clear account of any oversized-design or
  tool-capacity blocker.
- Failures must be structural and explicit, for example "op X has no legal
  lowering" or "memory schedule for tensor Y is not implemented". Failures must
  never appear as hidden placeholders or vague backend breakage.

** Deliverable priority and implementation constraints
- The most important Task 3 deliverable is the Yosys-based resource or
  bottleneck report.
- The project does not depend on =nextpnr-xilinx= being able to ingest the full
  generated design. If the emitted design is too large for =nextpnr-xilinx=,
  the flow may compute its own utilization report from synthesizable emitted
  hardware artifacts instead of relying on downstream place-and-route.
- This is a first-pass demonstration that there is a credible path for lowering
  LLMs to RTL through a standard open-source compiler stack.
- The allowed core toolchain is: =torch-mlir=, =mlir=, =circt=, and
  =yosys=/=yosys-slang=.
- The flow must not use =perl=, =sed=, or any other ad hoc text-rewrite tool to
  patch MLIR, SV, or other compiler IR artifacts inside the core path.
- The flow must not rely on non-standard, ad hoc preprocessing of PyTorch
  models. Frontend preparation must stay within standard, explainable PyTorch
  and torch-mlir-compatible mechanisms.
- The resulting SystemVerilog must be synthesizable. It may be too large to fit
  a specific FPGA, but final success must not depend on unresolved black boxes,
  unresolved stubs, or unsupported floating-point operations.
- The flake and reviewer docs must expose exactly one canonical path to the
  reviewer-visible Yosys result for Task 3. Parallel or competing Task 3 flows
  should be removed or clearly demoted so there is only one authoritative
  route.

** Scope boundary
- The Task 3 claim does not end at generated SystemVerilog. Task 3 must reach a
  Yosys handoff result through one canonical reviewer-visible flow built on
  synthesizable emitted SystemVerilog.
- The core lowering pipeline for the RfP must use only =torch-mlir=, =mlir=,
  and =circt= to reach SystemVerilog, followed by =yosys=/=yosys-slang= for the
  canonical synthesis/utilization path.
- =nextpnr-xilinx=, fasm, bitstream generation, and FPGA self-test wrappers are
  optional downstream validation steps. They are not required for Task 3
  success when the design is too large for downstream FPGA implementation tools
  to ingest.
- Temporary stubs may be used during debugging only if they are explicit and
  tracked. Reviewer-facing claims and final Task 3 success must identify them
  and must not count them as completed implementation.
- No sed/perl/text-rewrite surgery over model IR or emitted hardware is allowed
  inside the canonical Task 3 path.

** Explicit non-goal for now
- Do not add a generic SV floating-point operation library as the primary Task 3
  path.
- Rationale:
  1. it weakens the RfP story by moving the hard part to handwritten backend
     hardware
  2. it is likely expensive in area and verification effort
  3. it does not help establish a clean standard lowering path from PyTorch
