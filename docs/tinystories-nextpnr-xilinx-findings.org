** Confirmed nextpnr-xilinx Bug
The first ~nextpnr-xilinx~ failure was an immediate uncaught exception:
- ~std::out_of_range~
- ~vector::_M_range_check: __n (which is 2817) >= this->size() (which is 0)~

This occurred during JSON import, before pack/place/route.

*** Local root cause
In local ~nextpnr-xilinx~ source:
- file: ~frontend/frontend_base.h~

two parallel vectors were meant to stay in lockstep:
- ~net_flatindex~
- ~net_old_indices~

But new flat-indexed nets were appended to ~net_flatindex~ without appending a
corresponding entry to ~net_old_indices~. Later, ~merge_nets()~ indexed
~net_old_indices.at(mergee->udata)~, which crashed when the vector was still
empty.

*** Local patch
The local fix was to append an empty merge-history entry whenever a new
flat-indexed net is created:

- after ~net_flatindex.push_back(net);~
- after ~net_flatindex.push_back(cnet);~

by adding:

#+begin_src c++
net_old_indices.emplace_back();
#+end_src

This was patched in a local copy under:
- ~/tmp/nextpnr-xilinx-bugfix/frontend/frontend_base.h~

*** Result
That importer crash is real and locally fixed:
- before patch: immediate abort in JSON import
- after patch: importer gets past the old crash point and continues running

So there is at least one confirmed ~nextpnr-xilinx~ importer bug against this
TinyStories JSON.

*** Differential conclusion
The remaining importer blow-up is *not* explained mainly by:

Those all contribute some overhead, but the dominant memory growth still appears
to come from the actual imported hierarchical graph and its net/cell expansion.

** Best Current Interpretation
The current frontier is:
1. one real ~nextpnr-xilinx~ importer bug was fixed locally
2. TinyStories still causes importer-state growth large enough to reach
   ~27-28 GB~ RSS
3. most of that growth still happens before actual place-and-route

So the remaining problem is best treated as a ~nextpnr-xilinx~ scalability or
import-structure problem, not as a CIRCT or Yosys synthesis failure.

** Recommended Next Steps
1. Continue instrumenting ~nextpnr-xilinx~ importer state rather than changing
   CIRCT/Yosys again.
2. Focus on hierarchical import growth:
   - number of imported modules
   - number of duplicated cell attributes/params
   - per-module net/cell expansion
3. Test shorter Yosys-generated names before JSON export, because long
   auto-generated net names are now a confirmed importer hotspot.
4. Reduce remaining JSON metadata if possible, but do not expect that alone to
   solve the failure.
5. Investigate whether flattening only targeted sub-hierarchies near the top is

   flattening.
6. If a second concrete ~nextpnr-xilinx~ bug is found, patch it locally first
   against the frozen TinyStories JSON/XDC reproducer before moving back into

** Importer Instrumentation: Quantitative Breakdown
To move beyond single-stack ~gdb~ snapshots, local importer telemetry was added
to ~frontend_base.h~ around:
- ~import_module~
- ~import_module_netnames~
- ~create_or_get_net~

The instrumented binary was then run on:
- ~/tmp/tiny-stories-quant-int8-selftest-trimmed.json~

with stderr captured in:
- ~/tmp/nextpnr-instrumented.stderr~

*** Result
The run still died with:
- ~Command terminated by signal 9~
- max RSS: ~27,681,172 kB~ (~27.7 GB~)
- wall time: ~6:00.57~

*** Key Finding
The first-order blow-up happens in the top-level imported ~main~ instance
itself, before deep recursive hierarchy import dominates.

Early in the run, while still at:
- depth ~1~
- path ~tiny_stories_selftest_top/u_dut~
- type ~main~

the importer had already accumulated:
- ~local_bits=19,040,540~
- ~local_candidates=19,039,133~
- ~module_net_slots=19,040,542~
- ~local_aliases=1,407~

with RSS already around ~8.7 GB~, while actual created nets/cells were still
tiny by comparison:
- ~ctx_nets=3,184~
- ~ctx_cells=2,357~

Interpretation:
- the top-level ~main.netnames~ section is itself enormous
- the dominant early cost is storing per-bit candidate names in
  ~m.net_names~
- this is not initially a deep repeated-submodule problem

*** What Happens Next
After that initial ~main~ netname blow-up, recursive import of many
~handshake_buffer_*~ instances continues to push memory upward via alias/net
growth. Just before the kill, the instrumentation showed approximately:
- ~ctx_nets=11,495,767~
- ~ctx_cells=9,963,052~
- ~ctx_aliases=18,498,179~
- ~flat_nets=10,000,065~
- RSS ~27.5 GB~

So the import failure appears to have two stages:
1. huge top-level ~netnames~ candidate-name storage in ~main~
2. continued alias/net expansion during recursive import of many small modules

*** Consequence For Next Steps
The next highest-signal experiment is no longer "generic importer tuning". It
is to attack top-level ~main~ netname storage directly, for example by:
- not materializing all internal ~main~ netnames up front
- keeping top-level port names but synthesizing internal names lazily
- or rewriting the JSON so ~main.netnames~ is drastically reduced while
  preserving port/constraint visibility
