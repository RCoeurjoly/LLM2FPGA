Anti-churn rules:
- Do not edit docs, status notes, or non-build-facing cleanup files while the
  main gate is still failing, unless the change is strictly required for the
  build or to create a new minimal reproducer.
- Do not spend more than 3 consecutive top-level gate reruns on the same exact
  failing derivation and log line without either:
  1. a code change aimed at that blocker, or
  2. a strictly smaller reproducer that removes unrelated noise.
- Do not create a new reproducer unless it is expected to reduce the blocker
  within about 20 minutes. If it does not narrow the failure quickly, abandon
  it and return to the highest-value existing reproducer.
- Do not repeat long written summaries of the same blocker. Keep notes short and
  move on to implementation.
- Prefer patching code over restating status.

Required working loop:
1. Read the runtime state file if present and continue from the last unfinished
   next step instead of re-summarizing old work.
2. Reproduce the current top-level gate failure once if needed.
3. Identify the smallest failing derivation that still blocks the canonical
   path.
4. Work on that smallest blocker with fast local iteration.
5. Only rerun the top-level gate after a meaningful fix or a materially better
   reduction.
6. Update the runtime state file with:
   - current blocker
   - exact failing command
   - exact failing derivation or first log line
   - files changed
   - next concrete step
7. Resume from that next concrete step on the next pass.

Reproducer policy:
- Prefer existing reproducers over inventing new ones.
- A new reproducer is justified only if it makes the failing op or dialect
  boundary substantially clearer than the current best reproducer.
- If the blocker is already known precisely enough to patch, stop reducing and
  patch it.

Reporting rule:

  during an intermediate pass that may be resumed
- treat each pass as continuation-only unless you have either:
  1. succeeded on `nix build .#tiny-stories-1m-utilization -L`, or
