# ============================================================
# CORE PRINCIPLE
# ============================================================

A moonshot is NOT:
- vague futurism
- generalized optimism
- "AI will change everything"
- broad civilization narratives
- platform-building without a target problem

A moonshot IS:
- a specific massive problem
- attacked using a radical but physically plausible approach
- enabled by one or a few key breakthrough levers
- validated through aggressive de-risking and experimentation

Canonical structure:

1. Huge Problem
2. Radical Solution
3. Breakthrough Technology / Lever

This is called the Moonshot Blueprint.

# ============================================================
# MOST IMPORTANT IDEAS (ORDERED)
# ============================================================

# ------------------------------------------------------------
# 1. PROBLEM FIRST
# ------------------------------------------------------------

The most important rule.

Do NOT start with:
- technology
- AI
- robotics
- platforms
- architecture
- philosophy

Start with:
- a painful, massive, urgent problem

Good:
- "Construction is too expensive and slow."
- "Climate adaptation infrastructure cannot scale."
- "Current FPGA compilation flow is too resource intensive."

Bad:
- "Universal autonomous production."
- "AI-powered civilization."
- "Transforming everything."

If the problem statement is vague,
the project will fragment.

Symptoms of failure:
- too many application domains
- too many technologies
- unclear customer
- unclear first experiment
- unclear success criteria

# ------------------------------------------------------------
# 2. 10X THINKING
# ------------------------------------------------------------

Moonshots aim for:
- 10X improvement
NOT:
- 10% optimization

Reframing examples:

10%:
- reduce latency by 10%

10X:
- eliminate latency entirely

10%:
- make housing cheaper

10X:
- autonomous construction that reduces labor cost near zero

Key question:
"What assumption are we accepting that might be false?"

# ------------------------------------------------------------
# 3. BREAK ASSUMPTIONS
# ------------------------------------------------------------

List industry assumptions.
Flip them.

Examples:
- "Humans must assemble machines."
- "Factories must be centralized."
- "Repair requires humans."
- "Infrastructure must be permanent."

Question:
"What if this were false?"

Common transforms:
- zero cost
- zero waiting
- remove intermediary
- invert control
- automate maintenance
- make local/self-serve

# ------------------------------------------------------------
# 4. AXIOMS
# ------------------------------------------------------------

Axioms are strong beliefs about the future,
treated as true to guide decision-making.

Good axioms:
- specific
- directional
- operationally useful

Weak:
- "AI may become important."

Strong:
- "Within 10 years, AI-generated hardware designs will outperform most manually designed systems."

Purpose:
- remove ambiguity
- align teams
- guide prioritization

# ------------------------------------------------------------
# 5. MONKEY FIRST
# ------------------------------------------------------------

MOST IMPORTANT EXECUTION PRINCIPLE.

The "monkey" =
the hardest, riskiest, least certain part.

Test this FIRST.

DO NOT:
- optimize around the monkey
- postpone the monkey
- build infrastructure first

Examples:

Waymo monkey:
- reliable autonomous driving

LLM2FPGA monkey:
- FPGA board-fit + external memory reliability

SAMI likely monkeys:
- autonomous repair
- universal robotic assembly
- reversible modular architecture

A project without a clearly identified monkey
will diffuse into endless experimentation.

# ------------------------------------------------------------
# 6. PREMORTEM
# ------------------------------------------------------------

Assume the project failed.
Ask:
"Why?"

Purpose:
- fight overconfidence
- fight planning fallacy
- expose blind spots
- surface hidden objections

Scientific basis:
- anticipatory thinking
- debiasing
- prospective hindsight
- metacognition

Use before:
- roadmap commitment
- fundraising
- scaling

# ------------------------------------------------------------
# 7. RED TEAMING
# ------------------------------------------------------------

Deliberately attack the idea.

Find:
- hidden assumptions
- economic weaknesses
- technical impossibilities
- adoption barriers

Important:
criticism must be incentivized.

If criticism is socially punished,
the system becomes blind.

# ------------------------------------------------------------
# 8. KILL CRITERIA
# ------------------------------------------------------------

Define conditions under which the project stops.

Without kill criteria:
- sunk-cost fallacy dominates
- infinite roadmap expansion occurs

Good kill criteria:
- measurable
- objective
- technical or economic

Examples:
- "If DDR3 calibration reliability stays below X after Y experiments."
- "If autonomous repair requires excessive manual intervention."

# ------------------------------------------------------------
# 9. FAST & SCRAPPY PROTOTYPING
# ------------------------------------------------------------

Build the smallest thing
that maximizes learning.

This is NOT:
- polished product development

This IS:
- uncertainty reduction

Preferred:
- ugly
- cheap
- rapid
- informative

Goal:
"Make it real enough to feel."

# ------------------------------------------------------------
# 10. ITERATION SPEED
# ------------------------------------------------------------

Moonshot advantage often comes from:
- iteration loops
NOT:
- initial intelligence

Optimize:
- experiment cycle time
- simulation speed
- representative-core testing
- automation

Key principle:
small fast experiments > large slow experiments

# ------------------------------------------------------------
# 11. MARKET LISTENING
# ------------------------------------------------------------

Talk to:
- users
- critics
- outsiders
- operators
- maintainers
- regulators

Goal:
find repeated pain patterns.

Important:
people often describe symptoms,
not root problems.

# ------------------------------------------------------------
# 12. QUESTION BURSTS
# ------------------------------------------------------------

Generate questions only.
No answers.

Purpose:
- avoid premature convergence
- break solution fixation
- expose hidden dimensions

# ------------------------------------------------------------
# 13. BAD IDEA BRAINSTORMS
# ------------------------------------------------------------

Generate intentionally terrible ideas.

Purpose:
- remove perfectionism
- increase divergence
- bypass social inhibition

Often:
bad ideas contain hidden insights.

# ------------------------------------------------------------
# 14. INNOVATION SANDBOX
# ------------------------------------------------------------

Moonshots require protected environments.

Needed:
- permission to fail
- psychological safety
- long time horizons
- experimentation culture

Without this:
organizations optimize for local safety.

# ------------------------------------------------------------
# 15. COMFORT WITH AMBIGUITY
# ------------------------------------------------------------

Moonshots operate under:
- incomplete information
- uncertain timelines
- evolving architectures

Need:
- resilience
- adaptability
- probabilistic thinking

# ------------------------------------------------------------
# 16. PASSIONATELY DISPASSIONATE
# ------------------------------------------------------------

Care deeply about solving the problem.
Do NOT become emotionally attached
to a specific implementation.

Ability to kill your own ideas
is critical.

# ------------------------------------------------------------
# 17. STORYTELLING
# ------------------------------------------------------------

A moonshot story requires:

1. Massive problem
2. Why now
3. Radical solution
4. Plausible path
5. Human impact
6. Business value

Bad storytelling:
- abstract futurism
- generalized philosophy
- too many domains

Good storytelling:
- concrete
- focused
- emotionally legible
- technically plausible

# ------------------------------------------------------------
# 18. START WITH ONE DOMAIN
# ------------------------------------------------------------

Do NOT:
- solve everything simultaneously

Start:
- narrow
- specific
- painful
- high leverage

Examples:
- self-driving cars
- reusable rockets
- autonomous repair
- disaster recovery robotics

Generality can emerge later.

# ------------------------------------------------------------
# 19. SYSTEMS THINKING
# ------------------------------------------------------------

Moonshots often require:
- co-design
- hardware/software integration
- ecosystem optimization
- feedback loops

But:
systems thinking must NOT replace focus.

# ------------------------------------------------------------
# 20. UNIVERSALITY IS DANGEROUS
# ------------------------------------------------------------

"Universal platform" thinking
often causes failure.

Why:
- infinite scope
- no prioritization
- unclear monkey
- impossible roadmap

Preferred:
- constrained universality
- one strong wedge application

# ============================================================
# ANTI-PATTERNS
# ============================================================

# Vision-first instead of problem-first
# Platform before use-case
# Too many simultaneous breakthroughs
# No monkey identified
# No kill criteria
# Infinite roadmap expansion
# Philosophy replacing execution
# Generality before product-market fit
# Infrastructure before validation
# Lack of measurable milestones
# Premature ecosystem thinking

# ============================================================
# EVALUATING A MOONSHOT
# ============================================================

Checklist:

1. Is the problem concrete and painful?
2. Is the solution radical but physically plausible?
3. Is the breakthrough lever identifiable?
4. Is the monkey clearly defined?
5. Can the monkey be tested quickly?
6. Are there kill criteria?
7. Is there a first market/domain?
8. Is there a believable adoption path?
9. Does the project avoid scope explosion?
10. Are iteration loops optimized?

"one impossibly important thing,
